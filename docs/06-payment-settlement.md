# 06. 결제·정산

> ⚠️ **이 문서는 기술 설계 + 리스크 정리이며 법률 자문이 아니다.** 정산(플랫폼이 돈을 받아 작가에게 나눠줌)은
> 한국에서 규제 영역이다. 실제 정산 구조·약관은 **PG사 및 법무/세무 검토 후 확정**해야 한다.

## 왜 결제가 아니라 "정산"이 어려운가

단순 결제(유저 → 가맹점)는 쉽다. 문제는 **마켓플레이스 정산** — 플랫폼이 유저 결제를 일단 받고, 일정 조건(보정본 전달 확인) 후 **작가에게 송금**하는 구조다. 이때:

- 플랫폼이 자금을 보관·이체하면 **전자금융거래법상 자금이체/에스크로** 이슈가 생길 수 있다.
- 직접 정산을 구현하면 PG 가맹·세금계산서·원천징수·정산대행 등록 부담이 크다.
- **해결책: PG사의 "마켓플레이스/하위가맹점 정산" 솔루션을 사용**해 플랫폼이 직접 자금을 보관하지 않고 PG가 작가에게 분배하도록 위임한다.

## PG 후보 비교

| 후보 | 마켓플레이스 정산 | 특징 | 메모 |
|------|------------------|------|------|
| **포트원(PortOne, 구 아임포트)** | 여러 PG 통합 + 정산/하위가맹 연동 | 단일 SDK로 카드·간편결제 통합, webhook 표준화 | 통합·교체 유연. **1순위 검토** |
| **토스페이먼츠** | 마켓플레이스(하위 셀러 정산) 상품 제공 | 간편결제 UX 우수, 문서 양호 | 정산 상품 자격·심사 확인 필요 |
| **카카오페이/네이버페이** | 단독으론 정산 분배 제한적 | 간편결제 수단으로 포트원 경유 권장 | 결제수단으로 포함 |

**권장 접근**: 포트원으로 결제·수단을 통합하고, 정산은 PG의 하위가맹점/정산 기능 또는 정산대행을 활용. 자격·수수료·정산주기를 PG와 직접 협의해 [05](05-booking-lifecycle.md)의 흐름에 맞춘다.

## 자금 흐름 (에스크로식)

```
유저 결제 ──► PG ──► (보류/에스크로) ──► 전달 확인(completed) ──► 정산
                                                  │
                                    ┌─────────────┴─────────────┐
                                    ▼                           ▼
                              작가 수령액(net)            플랫폼 수수료(fee)
```

- 결제 시점: `accepted → paid`. PG webhook으로 확정.
- 보류 기간: `paid` ~ `completed`. 이 동안 환불 가능.
- 정산 시점: `completed` 후 정산 주기에 맞춰 `settlements.status` pending→scheduled→paid.

## 수수료 모델 (초안)

- 플랫폼 수수료 = 거래액 × 수수료율(예: 10~20%, 정책 확정 필요)
- PG 결제수수료는 수수료에서 흡수할지 작가에 전가할지 정책 결정
- `settlements`: `gross_krw`(결제총액) − `fee_krw`(플랫폼+PG) = `net_krw`(작가 수령)

## 환불

- 보류(`paid`~`delivered`) 중 환불은 PG 취소 API + `payments.status` 갱신 + 정산 취소
- 부분환불 지원(`refunded_krw` 누계)
- 환불 정책 수치는 [05](05-booking-lifecycle.md) 취소 정책과 동기화, 약관에 명시

## 구현 체크리스트 (5단계에서)

- [ ] PG 선정 및 가맹/정산 자격 신청
- [ ] 결제 사전등록 라우트 `/api/payments/prepare` — 서버가 금액 생성·보관(클라이언트 금액 신뢰 금지)
- [ ] 결제 webhook `/api/payments/webhook` — **서명 검증** + 서버 보관 금액 대조 + 멱등
- [ ] `bookings` 상태 전이와 트랜잭션 결합 (RPC)
- [ ] 정산 확정 배치/라우트 `/api/settlements/run`
- [ ] 환불 라우트 + 운영자 분쟁 처리 UI
- [ ] 감사로그(`payments.raw`, `settlements` 이력) + 멱등키 전수 적용
- [ ] 약관·환불정책·전자상거래법 고지 페이지

## 미해결 / 결정 필요 항목

1. **PG 최종 선정** (포트원 vs 토스페이먼츠) — 정산 자격·수수료 협의 후
2. **수수료율·정산주기** — 사업 정책
3. **정산 자금 보관 주체** — PG 위임 vs 자체(전자금융업 검토) → **위임 강력 권장**
4. **세금**: 작가 사업자/비사업자 구분, 원천징수·세금계산서 — v1은 수기 운영 후 자동화
5. **약관·개인정보처리방침·환불정책** 문서 (법무)
