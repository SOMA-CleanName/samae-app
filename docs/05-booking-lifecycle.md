# 05. 예약 생애주기 (상태머신)

`bookings.status`는 거래 전체의 **단일 진실**이다. 결제·정산·전달·후기·알림이 모두 이 상태를 보고 동작한다.
모든 상태 전이는 **서버 라우트/RPC에서만** 수행하며, 각 전이는 검증 → 부수효과 → 타임스탬프 기록 순으로 처리한다.

## 상태 다이어그램

```
                 ┌────────────┐
                 │ requested  │  유저가 예약 요청 (채팅 협의 후)
                 └─────┬──────┘
        작가 거절        │ 작가 수락
        ┌───────────────┼────────────────┐
        ▼               ▼                 │
   ┌─────────┐    ┌───────────┐           │
   │ rejected│    │ accepted  │ 금액·일시 확정 │
   └─────────┘    └─────┬─────┘           │
                        │ 유저 결제(PG paid)  │
                        ▼                  │ 결제 전 취소
                   ┌─────────┐             ▼
                   │  paid   │       ┌────────────┐
                   │(에스크로) │       │ cancelled  │
                   └────┬────┘       └────────────┘
                        │ 촬영 완료
                        ▼
                   ┌─────────┐
                   │  shot   │
                   └────┬────┘
                        │ 작가 보정본 업로드
                        ▼
                  ┌───────────┐
                  │ delivered │
                  └─────┬─────┘
       유저 전달 확인      │
         (또는 자동확인)    ▼
                  ┌───────────┐   정산 확정 → 작가 입금
                  │ completed │ ─────────────────────►
                  └───────────┘
                        │ 분쟁/하자
                        ▼
                  ┌───────────┐
                  │ refunded  │ (전체/부분)
                  └───────────┘
```

## 전이 규칙

| From → To | 트리거 | 검증 | 부수효과 |
|-----------|--------|------|----------|
| (none) → `requested` | 유저가 예약 요청 | 작가 승인됨·슬롯 존재·로그인 | 작가에게 알림, 대화방 연결 |
| `requested` → `accepted` | 작가 수락 | 본인 작가·금액/일시 설정 | `amount_krw`·`shoot_at` 확정 스냅샷, 슬롯 보류, 유저 알림 |
| `requested` → `rejected` | 작가 거절 | 본인 작가 | 유저 알림, 슬롯 해제 |
| `accepted` → `paid` | PG 결제 성공(**webhook**) | 금액 일치·멱등 | `payments` 생성/갱신, **에스크로 보류**, 양측 알림, 슬롯 확정 |
| `accepted` → `cancelled` | 결제 전 유저/작가 취소 | 결제 이력 없음 | 슬롯 해제, 알림 |
| `paid` → `shot` | 촬영 완료 표시 | 작가(또는 일시 경과) | 알림 |
| `shot` → `delivered` | 작가 보정본 업로드 | `deliveries` 생성·자산 존재 | 만료 다운로드 링크, 유저 알림 |
| `delivered` → `completed` | 유저 전달 확인 (또는 N일 후 자동확인) | 본인 유저 | **정산 확정**(`settlements` scheduled→paid), 후기 작성 가능, 알림 |
| `paid`/`shot`/`delivered` → `refunded` | 분쟁·하자·노쇼 | 운영자/규칙 | 환불(전체/부분), 정산 보류·취소, 감사로그 |

## 결제·정산과의 연동

- **결제는 `accepted → paid` 한 지점**에서만 발생. 신뢰의 원천은 **PG webhook** (클라이언트 콜백은 UX 표시용).
- 결제 즉시 작가에게 송금하지 않고 **플랫폼이 보류(에스크로)** → `completed` 시점에 정산 확정. (규제·흐름은 [06](06-payment-settlement.md))
- 자동 확인: `delivered` 후 일정 기간(예: 7일) 유저 무응답 시 자동 `completed` 처리 (작가 보호). 분쟁 신고가 있으면 보류.

## 취소·환불 정책 (초안 — 운영정책으로 확정 필요)

| 시점 | 정책(예시) |
|------|-----------|
| `accepted` (결제 전) | 위약 없음, 슬롯 해제 |
| `paid` 후 ~ 촬영 X일 전 | 전액 환불 |
| 촬영 임박(X일 이내) | 부분 환불(수수료 차감) |
| `delivered` 후 하자 | 운영자 분쟁 처리 → 부분/전액 환불 |
| 노쇼(작가/유저) | 귀책 측 패널티 (정책 필요) |

> ⚠️ 환불·위약 정책은 **약관·전자상거래법**과 직결. 수치는 운영/법무 확정 전까지 placeholder.

## 멱등성·정합성

- 모든 전이 API는 **현재 status를 조건으로 한 조건부 update**(낙관적 동시성)로 이중 처리 방지
- 결제 webhook은 `idempotency_key`/`pg_tx_id` unique로 중복 수신 방어
- 상태 전이 + 부수효과(payments/settlements/notifications)는 **하나의 트랜잭션** 또는 Postgres 함수(RPC)로 원자적 처리
