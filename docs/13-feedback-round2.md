### §A. 모바일 알림 — 결정: 인앱 알림만 (현행 유지)
- **현 구조**: `notifications` 테이블 + 인앱 벨/배지(완료). 단, **앱이 닫혀 있으면 도달 안 함**(한계 인지하고 보류).
- **외부 발송 보류 사유**:
  - **알림톡(최선이나 불가)**: 사업자등록 + 전화번호 수집 + 템플릿 심사 선행 → **현재 사업자등록 없음 → 착수 불가**.
  - **이메일(가능하나 약함)**: 도메인 인증 품 대비 한국 사용자 도달·확인율 낮음. 보강 효과 미미.
- **후속(사업자 확보 시)**: **알림톡을 메인**(거래 알림 3종: 수락·송금·전달) + 이메일 보조. SOLAPI 등 대행사 연동.
- **그 이후**: PWA Web Push(iOS 16.4+ 홈화면 추가 전제, 서비스워커+VAPID).

### §B. 보정본 전달 — 결정: 앱 내 전달 필수
- **목표 UX**: 작가가 "보정본 전달 완료"를 누르면 채팅방에 완료 메시지 + 후기 유도 카드. **사용자 수신확인 단계는 생략**(요구사항 → `deliveries.confirmed_at`은 자동/생략).
- **인프라**: `deliveries` 테이블 이미 존재(`asset_paths text[]`, `expires_at`). **신규 비공개 버킷 `samae-delivery` 필요**(기존 버킷은 public이라 부적합).
- **앱 내 전달(주)**: 작가가 비공개 버킷에 보정본 업로드 → `asset_paths`에 경로 저장 → 사용자는 **서명 URL(만료부)** 로만 다운로드(구매자 RLS 게이트).
  - 용량: Supabase Free 1GB / Pro 100GB. 고화질 30장 ≈ 수백 MB → 보관기간(`expires_at`, 예: 30일) 두고 만료 정리.
- **외부 링크(보조)**: 대용량/RAW는 작가가 드라이브 링크를 카드에 첨부하는 옵션도 허용.

### C3 송금 카드 — 상태 전이 정리
```
accepted(수락)
  → 채팅방: [송금 안내 카드] (작가 계좌·금액·정책) + [송금 완료] 버튼
  → 사용자 [송금 완료] 클릭
  → status: paid  +  작가에게 알림("입금했어요")
  → 작가가 입금 확인 (수동) … (별도 확인 액션 검토)
```
> 직접 이체 모델([06 결제·정산](06-payment-settlement.md))이라 **앱은 송금을 보증하지 않음** → 정책 카드에서 분쟁/환불 한계를 명시해야 신뢰 공백(12-UX #3) 완화.
